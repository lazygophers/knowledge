# 库表设计详解

## 新手：最低可行设计法 {#beginner}

### 先回答 4 个问题

1. 这张表最核心的查询路径是什么？
2. 哪个字段是天然主键，哪个字段只是展示信息？
3. 哪些字段允许缺失，缺失时业务语义是什么？
4. 哪些约束必须由数据库兜底，哪些可以只在应用层校验？

### 最小检查清单

- 主键策略已确定：自增 / 分布式 ID / 业务主键。
- 唯一约束覆盖了真正不能重复的数据。
- 时间字段、状态字段、金额字段的默认值语义清晰。
- 高频查询字段已经对应索引或明确说明暂不建立索引。

## 进阶：约束、索引与一致性协同 {#intermediate}

### 1. 先有查询，再有索引

索引应该服务于稳定的访问模式，而不是“字段看起来常用就先建上”。设计时至少明确：

- 过滤条件是什么；
- 排序字段是什么；
- 是否需要覆盖查询；
- 写入频率是否会让索引维护成本失控。

### 2. `NULL` 与默认值是业务契约

- `NULL` 表示未知、缺失、尚未发生，还是“不适用”？必须统一。
- 默认值会影响统计、排序、聚合和幂等逻辑，不能只图省事。
- 一旦默认值被当成“正常数据”，后续清洗和补数会非常昂贵。

### 3. 唯一约束决定写入失败路径

唯一键不仅用于防重，还定义了：

- 幂等请求如何判重；
- 冲突时返回什么错误；
- 补偿任务如何避免重复落库；
- 业务键变更时如何迁移历史数据。

## 高级：演进、容量与评审推演 {#advanced}

### 容量与冷热分层

- 大字段和热点字段混在一起，会导致缓存、页命中和回表都变差。
- 明显低频的说明类字段、扩展 JSON、审计信息，可以考虑拆到副表或归档表。
- 不是所有拆表都值得做，先验证热路径是否真的被大字段拖垮。

### 索引成本不是免费午餐

每增加一个索引，都要问：

- 写入路径会多多少维护成本？
- 是否增加 page split 或 buffer pool 压力？
- 这个索引是否和其他索引重复？
- 当业务查询变化时，它会不会变成“没人删的遗留索引”？

### 评审要问的 6 个问题

1. 查询路径是否已经足够稳定？
2. 字段语义是否能支撑未来 2~3 次迭代？
3. 约束是否覆盖了真正的数据一致性边界？
4. 索引是否与排序、范围查询、覆盖查询协同？
5. 大字段、审计字段、冷热数据是否需要分层？
6. DDL 变更、回填、回滚方案是否已经可执行？

## 落地建议

- 设计文档里同时给出字段表、查询样例、索引设计和风险说明。
- 建表评审不要只看 ER 图，要把典型 SQL 一起拿出来过一遍。
- 任何“以后再补索引”的决定，都应该附带监控与回收计划。
