# 库表设计

这一主题聚焦 **建模结构、字段语义、约束策略、索引协同与后续演进**。如果你面对的是“新表怎么建、老表怎么改、为什么这个索引总不稳定”这类问题，从这里开始。

## 适合解决什么问题

- 新业务建表，不知道如何定义主键、唯一约束与字段默认值。
- 表已经上线，担心后续字段扩展、热点读写或索引回收成本。
- 设计评审时，需要把“业务语义”和“数据库约束”对齐。

## 新手：先把表建对 {#beginner}

- 先写清楚**最重要的查询场景**，再决定字段和索引。
- 主键只做一件事：稳定标识一行数据，不要混入易变业务含义。
- `NULL`、默认值、唯一约束必须和业务语义一致，否则后续统计与补数会很痛苦。
- 大字段、低频字段尽量和高频热点路径分离，避免无意义 I/O。

## 进阶：理解机制与约束边界 {#intermediate}

- 字段类型选择会直接影响行长度、索引体积和排序比较成本。
- “唯一”不只是防重复，还定义了写入失败与幂等处理方式。
- 索引不是事后补丁，应该和查询形态、排序需求、更新成本一起设计。
- 反范式要有明确收益场景，否则先保证一致性和可维护性。

## 高级：做生产级演进设计 {#advanced}

- 设计前就考虑字段扩展、冷热分层、历史归档和未来拆分路径。
- 评估每个索引的收益与维护成本，避免“查询快了，写放大和缓存命中却崩了”。
- 在线 DDL、回填脚本、主从同步延迟都应该在设计阶段被提前推演。
- 评审时不仅问“现在能不能跑”，还要问“半年后能不能继续演进”。
