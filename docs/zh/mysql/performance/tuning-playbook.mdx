# 调优作战手册

## 新手：最小有效排查法 {#beginner}

### 先抓三样东西

1. 最慢或最重的 SQL 样本。
2. 这条 SQL 的执行计划。
3. 它对应的业务场景与访问频率。

### 不要一开始就做的事

- 不要先加机器，再去猜 SQL 是否有问题。
- 不要看到某个字段慢，就条件反射给它单列索引。
- 不要只看一条样本 SQL，而忽略整类请求的分布。

## 进阶：用 EXPLAIN 和索引治理定位问题 {#intermediate}

### 调优的推荐顺序

1. 明确业务目标：这条 SQL 为谁服务，延迟目标是什么？
2. 看执行计划：扫描行数、回表、排序、临时表是否合理？
3. 看索引：是否命中过滤条件、排序条件、覆盖场景？
4. 看 SQL 形态：是否能减少不必要列、排序、子查询或范围扫描？
5. 最后才讨论缓存、拆分、异步化等架构动作。

### 复合索引常见误区

- 顺序错了，索引存在也未必真正帮到查询。
- 同时服务多个查询时，可能需要权衡而不是试图“一把梭”。
- 覆盖索引能减少回表，但索引体积变大也会带来维护成本。

## 高级：验证、回归与稳定性 {#advanced}

### 每次调优都要回答

1. 这次优化改善了哪个瓶颈？
2. 会不会让写入更慢、缓存命中更差或锁冲突更高？
3. 数据量继续上涨后，执行计划还稳不稳？
4. 如果效果不好，回滚动作是什么？

### 建议的验证闭环

- 调优前记录基线：延迟、QPS、扫描行数、资源占用。
- 调优后在相似负载下重跑验证。
- 把收益、代价、适用条件写回知识库，避免团队重复踩坑。

## 常用检查清单

- 已定位到真实最重 SQL，而不是体感最慢页面。
- 执行计划、索引设计、查询目标三者一致。
- 调优动作已经过回归验证。
- 有明确回滚方案和上线窗口控制。
