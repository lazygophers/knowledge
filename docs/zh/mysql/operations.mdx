# 运维检查

这一主题覆盖 **发布前检查、运行中观测、备份恢复、事故处置与复盘沉淀**。目标不是列更多清单，而是把“上线、运行、出事”三个阶段串成一条可执行链路。

## 适合解决什么问题

- 发布前不知道该验证哪些风险点。
- 线上指标很多，但不知道哪些真正决定数据库健康。
- 做了备份，却没验证过恢复路径。
- 事故处理结束后，知识没有沉淀回团队流程。

## 新手：先有一张能执行的检查单 {#beginner}

- 发布前确认备份可恢复、DDL 锁影响已评估、关键 SQL 已验证。
- 运行中至少盯住连接数、延迟、TPS、错误率和复制延迟。
- 事故后不要只口头复盘，要把过程回填成标准动作。

## 进阶：把观测与变更串起来 {#intermediate}

- 运维不是单独阶段，而是设计、发布、运行之间的闭环。
- 监控指标需要和数据库角色、业务窗口、容量阈值一起解释。
- 变更发布前后的观测面必须一致，否则回归判断没有依据。
- 备份策略只有在恢复演练通过之后才算真正可用。

## 高级：为事故与扩容做推演 {#advanced}

- 高级治理的重点是恢复能力，而不是“平时看起来没问题”。
- 需要提前定义容量阈值、降级动作、切流策略和回滚窗口。
- 事故处理不仅要恢复服务，还要沉淀检测、响应和预防机制。
- 任何例行检查都要能映射到真正的风险：数据丢失、长时间锁表、复制延迟、容量失控。
