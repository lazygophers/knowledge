# 事务与锁

这一主题关注 **一致性边界、锁冲突、死锁排查与重试策略**。如果你的系统已经开始出现锁等待、热点更新、长事务或难以复现的并发 bug，这里是入口。

## 适合解决什么问题

- 不确定该选 RC、RR 还是更严格的隔离级别。
- 线上出现锁等待、死锁，日志看不懂。
- 事务逻辑越来越长，不知道如何缩短持锁时间。
- 幂等、重试、补偿与事务边界互相打架。

## 新手：先避免明显踩坑 {#beginner}

- 事务越短越好，只包住真正需要一致性的 SQL。
- 更新同一批资源时，访问顺序保持一致，能显著减少死锁概率。
- 出现死锁先取证，再重试，不要靠拍脑袋加锁。
- 如果一条 SQL 没走索引，它持有和扫描的锁范围往往会超出你的直觉。

## 进阶：理解锁为什么会打架 {#intermediate}

- 锁行为取决于隔离级别、SQL 形态、是否命中索引和范围条件。
- 死锁不是“数据库坏了”，而是不同事务形成了循环等待。
- 一致性要求、吞吐量和重试成本是一个三角形，不能只优化其中一角。
- 幂等写入与事务边界设计必须一起看，否则重试会放大副作用。

## 高级：并发推演与恢复策略 {#advanced}

- 为热点业务设计事务时，要同时考虑持锁时长、回滚成本和失败重试风暴。
- 锁问题的真正治理，通常来自访问顺序、索引设计和热点拆解，而不是一味加超时。
- 高并发下要验证事务重试是否会造成流量放大、缓存击穿或消息重复消费。
- 线上事故处理要能在“降级、限流、拆流量、回滚”之间快速切换。
