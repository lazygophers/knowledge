# 锁机制与诊断

## 新手：先记住排查顺序 {#beginner}

### 看到锁问题先做这三步

1. 用 `SHOW ENGINE INNODB STATUS` 或等价监控拿到死锁/锁等待现场。
2. 找到冲突 SQL，确认是否命中正确索引。
3. 检查事务范围、访问顺序和是否有长事务占锁。

### 最容易犯的错

- 把大量读写逻辑放进一个事务里。
- 认为“按主键更新就一定不会死锁”。
- 没有索引还做范围更新，导致锁范围扩大。

## 进阶：隔离级别、锁类型与索引命中 {#intermediate}

### 隔离级别不是越高越好

- `RC` 更偏向吞吐和较低锁冲突。
- `RR` 在 InnoDB 中经常是默认选择，但范围查询与间隙锁要格外小心。
- 更高隔离级别会带来更强的一致性语义，但也会提高冲突和等待成本。

### 为什么索引决定锁行为

如果 SQL 能精准命中索引，锁通常更聚焦；如果退化成大范围扫描：

- 扫描更多记录；
- 持有更大范围的锁；
- 更容易和其他事务形成交叉等待。

### 死锁治理的正确目标

目标不是“彻底消灭死锁”，而是：

- 降低死锁概率；
- 缩短恢复时间；
- 确保失败请求可安全重试。

## 高级：生产环境的并发推演 {#advanced}

### 设计事务时要一起回答

1. 哪一步真正需要 ACID 一致性？
2. 哪一步可以拆成异步或补偿？
3. 失败后重试是否会放大热点？
4. 事务回滚会不会拖垮高峰流量？

### 常见治理动作

- 固定访问顺序，减少循环等待。
- 缩短事务范围，把外部调用移出事务。
- 用正确索引缩小锁范围，而不是事后补应用层互斥。
- 为死锁和锁等待设计明确的重试、告警和人工介入阈值。

## 诊断检查清单

- 冲突 SQL 已定位并复现。
- 执行计划确认索引命中正确。
- 长事务来源、持锁时间、调用链都能观测。
- 重试策略有上限，并已验证幂等与副作用边界。
