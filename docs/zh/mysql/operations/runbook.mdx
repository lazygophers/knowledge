# 运行维护手册

## 新手：最小可执行运维动作 {#beginner}

### 发布前

- 备份已完成，且至少验证过一次恢复流程。
- DDL 变更已评估是否会锁表、回填耗时、影响复制延迟。
- 关键慢 SQL 已压测或通过历史数据验证。

### 运行中

- 连接数、TPS、平均/尾延迟、错误率、主从延迟有可视化。
- 告警有分级，值班人知道什么情况下必须人工介入。

### 事故后

- 记录影响范围、恢复动作、根因、后续修复项。
- 把这次案例沉淀进知识库，而不是只留在聊天记录里。

## 进阶：把变更、监控、恢复联动起来 {#intermediate}

### 为什么只做 checklist 不够

因为数据库风险常常来自链路联动：

- 发布动作改变了 SQL 形态；
- SQL 形态改变了索引命中和锁行为；
- 锁行为再反过来放大延迟和复制压力。

所以运维检查必须和 schema、事务、性能三块知识联动。

### 建议的日常治理动作

- 周期性查看容量增长、慢 SQL TopN、复制延迟波动。
- 对高风险变更建立预检查模板和回滚模板。
- 为恢复动作准备演练记录，而不是只保留脚本文件。

## 高级：恢复能力与事故推演 {#advanced}

### 事故治理要回答的 4 个问题

1. 服务如何最快恢复？
2. 数据一致性是否受损？
3. 是否需要限流、切流、降级或回滚？
4. 下次如何更早发现、缩小影响？

### 高级检查重点

- 备份恢复的耗时是否满足业务窗口。
- 高峰期扩容或切换是否有演练记录。
- 监控是否能区分“数据库慢”与“应用打满数据库”。
- 事故后的知识沉淀是否进入评审、发布和巡检流程。

## 推荐沉淀模板

- 变更背景与目标
- 风险假设与验证结果
- 故障征兆与首个报警点
- 临时止损动作与最终修复动作
- 需要补齐的监控、脚本、流程、知识库页面
