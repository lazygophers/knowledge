## 新手：先把链路跑通 {#beginner}

- 建表前先写清楚**核心查询路径**，再决定字段和索引。
- 事务先做到**范围小、顺序稳、失败可重试**。
- 性能问题先找**最重 SQL**，别一上来就怀疑架构。
- 发布前确保**备份可恢复、监控可观测、DDL 锁影响已评估**。

## 进阶：理解机制与边界 {#intermediate}

- 为什么 `NULL`、唯一约束、默认值会影响业务语义和统计结果。
- 为什么 InnoDB 的锁冲突常常不是“加了锁”这么简单，而是与索引命中、访问顺序、隔离级别耦合。
- 为什么慢 SQL 的优化顺序通常是：业务目标 → SQL 形态 → 索引 → EXPLAIN → 验证。
- 为什么运维治理要同时覆盖发布、运行、事故三个时段，而不是只盯报警面板。

## 高级：做生产级推演与权衡 {#advanced}

- 设计一张表时，要同时考虑**写放大、冷热分层、索引维护成本、未来演进**。
- 处理事务问题时，要能推演**一致性要求、并发热点、回滚成本、重试策略**。
- 做性能治理时，要从**资源瓶颈、执行计划稳定性、回归验证**三方面闭环。
- 做运行维护时，要准备**容量阈值、备份恢复演练、变更回滚方案、事故复盘模板**。
