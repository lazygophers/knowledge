# 性能优化

这一主题解决 **慢 SQL、执行计划不稳、索引设计失衡、调优顺序混乱** 这些典型问题。重点不在“堆技巧”，而在建立可重复的分析闭环。

## 适合解决什么问题

- 不知道该先看 SQL、索引、还是机器资源。
- EXPLAIN 看了很多次，但还是不知道该怎么改。
- 索引越来越多，写入变慢，查询却没明显变快。
- 调优之后缺少验证和回归机制。

## 新手：先按正确顺序处理 {#beginner}

1. 先定位最重、最慢、最常执行的 SQL。
2. 再确认执行计划和索引命中是否合理。
3. 然后再考虑 SQL 改写、字段调整或架构扩展。

只要顺序错了，就容易陷入“机器先扩了，问题还在”的假优化。

## 进阶：建立分析闭环 {#intermediate}

- 慢 SQL 不只看单次耗时，还要看调用频率、影响范围和业务窗口。
- EXPLAIN 的价值在于帮助你理解访问路径，而不是机械追求某个字段好看。
- 覆盖索引、复合索引、排序字段布局都需要和查询模式一起看。
- 调优动作必须带验证，否则很容易把性能问题从读路径转移到写路径。

## 高级：资源与计划稳定性权衡 {#advanced}

- 真正困难的不是“偶发慢”，而是执行计划在数据量变化后变得不稳定。
- 调优要同时看 CPU、I/O、内存、锁等待和缓存命中，而不是只盯 SQL 文本。
- 每个索引和改写动作都应该有收益预期、回归验证和回滚路径。
- 高级优化的重点是确定“最值得优化的瓶颈”，而不是把所有 SQL 都抠一遍。
